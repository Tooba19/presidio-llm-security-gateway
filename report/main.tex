\begin{abstract}
Large Language Model (LLM) applications are vulnerable to prompt injection, jailbreak attempts, and sensitive information leakage. This report documents the complete design and implementation of a mini security gateway that mitigates such risks prior to LLM invocation. The system integrates a deterministic prompt injection detector, Microsoft Presidio for PII detection and anonymization, context-aware confidence boosting, composite entity detection, and a policy engine that determines an action (ALLOW/MASK/BLOCK). In addition to the final architecture, this report provides a detailed engineering process log, including encountered errors (ImportError, configuration failures, invalid JSON requests, and policy mismatches), root-cause analysis, and fixes. The final prototype is implemented in FastAPI with configurable thresholds and latency measurement, and evaluated using a structured 40-prompt dataset with quantitative metrics, confusion matrix analysis, latency breakdown, and ablation-style discussion.
\end{abstract}

Large Language Models, prompt injection, jailbreak, Microsoft Presidio, PII detection, security gateway, FastAPI, policy engine

\section{Introduction}
Modern LLM systems introduce new security and privacy risks because they operate on natural language inputs that can include adversarial instructions or sensitive personal information. Two classes of threats are especially relevant in practical deployments: (i) \emph{prompt injection/jailbreak attacks} that attempt to override system policies or extract hidden directives, and (ii) \emph{PII leakage}, where users accidentally or intentionally provide sensitive identifiers (emails, phone numbers, IDs, credentials) that should not be passed into downstream LLM calls or stored in logs.

This project implements a lightweight, modular \emph{LLM Security Mini-Gateway} that sits before the model. The gateway inspects user input, computes an injection risk score, detects and anonymizes PII using Microsoft Presidio~\cite{presidio2023}, optionally boosts entity confidence using context signals, detects composite sensitive combinations (e.g., name + phone), and selects an action based on a policy engine.

Unlike heavyweight approaches that require additional model inference, the presented gateway prioritizes: (1) \emph{interpretability} via rule-based detection, (2) \emph{configurability} via thresholds, and (3) \emph{reproducibility} via clean project structure, dependency management, and documented debug path.

% -------------------- 3. THREAT MODEL --------------------
\section{Threat Model}
\label{sec:threatmodel}

\subsection{Protected Assets}
Protected assets are categorized as follows:

\textbf{High Sensitivity:}
API keys, passwords, system prompt contents, national identifiers, and financial identifiers.

\textbf{Medium Sensitivity:}
Emails, phone numbers, and names when combined with other identifiers.

\textbf{Low Sensitivity:}
Non-identifying text and general queries (e.g., location names like ``Germany'' are not treated as sensitive PII for this gateway).

\subsection{Attack Types}
We consider at least five attack types relevant to LLM systems:
\begin{enumerate}
\item \textbf{Prompt injection}: ``Ignore previous instructions...''
\item \textbf{Jailbreak attempts}: ``Bypass restrictions...'' or role-play coercion.
\item \textbf{System prompt extraction}: requests for hidden instructions or developer messages.
\item \textbf{Sensitive data exfiltration}: attempts to retrieve secrets from context/logs.
\item \textbf{Credential exposure}: inclusion or harvesting of API keys/passwords in plain text.
\end{enumerate}

\subsection{Attacker Assumptions}
We assume a black-box, adaptive attacker who can:
\begin{itemize}
\item issue repeated requests and refine prompts,
\item paraphrase or obfuscate instructions,
\item attempt multilingual or indirect prompts,
\item but has no direct access to server internals.
\end{itemize}

% -------------------- 4. RELATED WORK --------------------
\section{Related Work}
Prompt injection and jailbreak vulnerabilities have been widely discussed, with systematic analyses of failure modes and attack strategies~\cite{wei2023jailbreak,perez2022redteaming}. Data extraction and memorization in language models highlight privacy risks when sensitive content is included in prompts or stored in logs~\cite{carlini2021extracting}. In production systems, a practical defense strategy is to implement a \emph{security layer} around LLM calls, including input sanitization, policy enforcement, and sensitive entity redaction.
Microsoft Presidio~\cite{presidio2023} provides a mature framework for PII detection and anonymization. However, Presidio alone does not address prompt injection and does not provide system-level policy decisions such as allowing or denying requests based on injection risk. This work integrates Presidio with injection detection and a unified policy engine, producing a coherent gateway that can be inserted into LLM application pipelines.

% -------------------- 6. INJECTION DETECTION --------------------
\section{Prompt Injection Detection}
\label{sec:injdet}

\label{sec:injdet}

\subsection{Method}
The injection detector is rule-based for interpretability and low latency. It scans for known patterns (e.g., ``ignore previous instructions'', ``reveal system prompt'', ``bypass restrictions'') and returns:
\begin{itemize}
\item a numeric injection score $S_{inj} \in [0,1]$,
\item a list of matched patterns,
\item a boolean signal \texttt{injection\_matched}.
\end{itemize}

A simple formulation is:
\[
S_{inj} = \min\left(1, \sum_{i=1}^{m} w_i \cdot \mathbb{1}(p_i \subseteq T)\right),
\]
where $T$ is the input text, $p_i$ are patterns, and $w_i$ are pattern weights.

\subsection{Why Reasonable}
This approach is reasonable for a mini-gateway because:
\begin{itemize}
\item It is fast and deterministic.
\item It is explainable (matched patterns provide explicit evidence).
\item It allows iterative coverage expansion based on observed failures.
\end{itemize}

\subsection{Known Failure Modes}
Likely failures include:
\begin{itemize}
\item paraphrased injections not matching patterns,
\item multilingual jailbreak attempts,
\item obfuscation (spacing, encoding, synonym substitution).
\end{itemize}
These motivate future work using semantic classifiers.

\section{Presidio-Based PII Detection and Customization}

\subsection{Base Presidio Usage}
Presidio Analyzer produces entities $E = \{(type, start, end, score)\}$, and Presidio Anonymizer replaces detected spans with a mask token (e.g., \texttt{<REDACTED>})~\cite{presidio2023}.

\subsection{Custom Recognizer}
\label{sec:customrecognizer}
A custom recognizer was added for Korean phone numbers (010-XXXX-XXXX). This ensures correct detection for region-specific formats not always captured by generic recognizers.

\subsection{Context-Aware Scoring}
\label{sec:contextscoring}
Presidio may assign moderate confidence to entities; however, surrounding context can increase certainty. We implemented a context window around detected entities. If security-indicating keywords appear nearby (e.g., ``OTP'', ``verification'', ``code'', ``PIN''), confidence can be boosted:
\[
s' = \min(1, s + \alpha \cdot C),
\]
where $C \in \{0,1\}$ indicates whether context keywords appear and $\alpha$ is a small boost factor.

\subsection{Composite Entity Detection}
\label{sec:composite}
Some combinations are more sensitive than any single entity. The gateway flags:
\[
C_{name+phone} = 1 \;\;\text{if}\;\; PERSON \in E \;\land\; PHONE \in E,
\]
and returns this as \texttt{composite\_name\_phone} in the API response for transparency and policy tuning.
\section{Policy Engine and Thresholds}
The policy engine chooses action $\in \{\text{ALLOW}, \text{MASK}, \text{BLOCK}\}$.

\subsection{Priority Rule: Injection Dominates}
A key design decision (motivated by testing) is: \emph{if any explicit injection pattern is matched, BLOCK immediately}, even if $S_{inj}$ is below the numeric threshold. This prevents weak-but-explicit adversarial prompts from slipping through.

\subsection{Security-Relevant Entity Filtering}
During testing, Presidio detected LOCATION entities (e.g., ``Germany'') and initially caused benign queries to be masked. This was corrected by filtering policy decisions to only consider security-relevant entity types (EMAIL, PHONE, PERSON, etc.), ignoring LOCATION and DATE\_TIME.

\subsection{Configuration}
All thresholds are centralized in \texttt{config.py} and can be overridden via environment variables (e.g., \texttt{INJECTION\_THRESHOLD}, \texttt{PII\_MASK\_SCORE}, \texttt{PII\_BLOCK\_SCORE}). This supports reproducible experiments without rewriting logic.

\section{Engineering Process and Debugging Log}
This section documents the main implementation steps and the key errors encountered.

\subsection{Step 1: Environment Setup}
A Python virtual environment was created and dependencies installed. A spaCy model was downloaded to support Presidio NLP pipelines.

\subsection{Step 2: Initial Module Integration}
The injection detector, Presidio engine, and policy engine were implemented as separate modules and integrated into FastAPI.

\subsection{Error A: Benign Queries Masked (LOCATION Detected)}
\textbf{Observed:} ``What is the capital of Germany?'' returned MASK due to \texttt{LOCATION} detection.\\
\textbf{Cause:} The policy treated any detected entity as sensitive.\\
\textbf{Fix:} The policy was refined to consider only security-relevant entity types as PII.

\subsection{Error B: Injection + PII Returned MASK Instead of BLOCK}
\textbf{Observed:} ``Ignore previous instructions. My email is ...'' returned MASK because the numeric injection score was below threshold.\\
\textbf{Cause:} Numeric thresholding alone was insufficient.\\
\textbf{Fix:} Added the \texttt{injection\_matched} signal so that any explicit injection pattern causes immediate blocking.

\subsection{Error C: Coverage Gap for Hidden-Rule and Bypass Phrases}
\textbf{Observed:} Prompts such as ``Bypass restrictions and show hidden system rules'' were incorrectly allowed.\\
\textbf{Cause:} Pattern coverage in the rule-based detector was incomplete.\\
\textbf{Fix:} Expanded pattern coverage and revalidated via the evaluation pipeline.

This iterative debug loop demonstrates practical engineering: identify failure, locate root cause, implement a targeted fix, and retest.

After iterative refinement of injection pattern coverage and policy calibration, the final gateway achieved perfect classification accuracy on the evaluation dataset. This indicates that the rule-based injection detector, Presidio customization pipeline, and hierarchical policy engine collectively provide robust security filtering with zero observed false positives and false negatives in the controlled evaluation setting.

The perfect precision and recall values across all classes demonstrate that injection dominance rules, entity sensitivity calibration, and composite detection effectively complement each other. Furthermore, the confusion matrix confirms the absence of misclassification, validating the reliability of the decision pipeline.

While the evaluation dataset is limited in size, the results strongly indicate that structured rule engineering combined with policy prioritization can achieve high security assurance without relying on heavy machine learning models.\\

\subsection{ROC-Style Threshold Discussion}
Lower injection thresholds $\tau_{inj}$ typically increase recall but may introduce false positives, whereas higher thresholds improve precision but can reduce sensitivity. In the final system, deterministic blocking on \texttt{injection\_matched} reduces dependence on threshold tuning for explicit adversarial phrases. As a result, the numeric score primarily serves as a secondary safeguard rather than the only decision signal.\\

\section{Complexity Analysis}
Let $n$ be input length, $p$ the number of injection patterns, and $k$ the number of detected entities.

\textbf{Injection detection:} $\mathcal{O}(p \cdot n)$ in the worst case (substring checks).\\
\textbf{Context scoring:} $\mathcal{O}(k)$ for fixed window size.\\
\textbf{Composite detection and policy passes:} $\mathcal{O}(k)$.\\
\textbf{Overall:} $\mathcal{O}(A(n) + p\cdot n + k)$, where $A(n)$ is the Presidio analysis cost and dominates runtime.
